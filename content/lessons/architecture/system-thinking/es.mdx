# Arquitectura de frontend

```tsx
// UI separada de datos: props claras, sin leer store global adentro
function UserCard({ user, onEdit }: { user: User; onEdit: () => void }) {
  return (
    <article>
      <h2>{user.name}</h2>
      <button type="button" onClick={onEdit}>
        Editar
      </button>
    </article>
  );
}
```

**Pensar en sistemas** es decidir dónde viven las fronteras: entre features, entre capas (UI / dominio / datos) y entre equipos. Componentes con contratos explícitos (`props`) son más fáciles de mover o testear que los que “leen el mundo” desde cualquier lado.

## Acoplamiento

```ts
// Alto acoplamiento: módulo A importa detalles internos de B
import { internalCache } from "../featureB/cache";

// Menor acoplamiento: interfaz estable
import type { CartRepository } from "./ports";
export function checkout(cart: Cart, repo: CartRepository) {
  return repo.charge(cart);
}
```

Cuanto más cosas “saben” unas de otras, más caro es cambiar una pieza sin romper otras. Las interfaces claras (props, hooks, servicios) reducen ese acoplamiento: dependés de contratos, no de implementaciones.

```ts
// Duplicación pequeña y local a veces vence a una abstracción prematura
function formatMoneyEs(n: number) {
  return new Intl.NumberFormat("es-AR", { style: "currency", currency: "ARS" }).format(n);
}
```

<Callout title="Prematura optimización">
No todo debe ser “genérico” desde el día uno: a veces un poco de duplicación honesta es mejor que abstracciones frágiles.
</Callout>

## Señales de salud del sistema

Más allá del código, mirá **métricas de producto y plataforma**: tasa de error por release, tiempo hasta interactivo, ratio de éxito de pagos o flujos críticos. Un buen diseño modular debería permitir aislar qué módulo empeoró cuando un gráfico salta.
