# Tests en el frontend

```ts
import { sum } from "./math";

it("suma positivos", () => {
  expect(sum(2, 3)).toBe(5);
});
```

Los tests automatizados reducen el miedo a **cambiar código**: si rompés algo importante, un test debería fallar antes de llegar a producción. El ejemplo es un test unitario mínimo sobre una función pura.

## Pirámide práctica

```ts
// Unit: función pura
export function discount(price: number, pct: number) {
  return price * (1 - pct / 100);
}

// Integration: varios módulos (p.ej. hook + formatter)
// E2E: Playwright/Cypress abre el browser y hace click real
```

Muchos equipos combinan **tests unitarios** (funciones puras), **integración** (varios módulos juntos) y **E2E** (flujo real en browser) según costo y valor: más arriba en la pirámide, más lento y más caro de mantener.

```js
// Ejemplo frágil: acoplado a clase CSS generada
screen.getByClassName("sc-12ab");
```

<Callout title="Tests frágiles">
Si un test depende de tiempos, orden de red o selectores CSS muy volátiles, se vuelve **flaky** (a veces pasa, a veces no) y pierde credibilidad.
</Callout>

## Coste y ritmo

Escribir y mantener tests tiene **coste fijo**: fixtures, datos de prueba, tiempo de CI. El retorno aparece cuando **refactorizás con confianza** o cuando un test evita un bug que hubiera costado horas en soporte. Ajustá la inversión al riesgo del módulo: dominio financiero ≠ icono decorativo.
