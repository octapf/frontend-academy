# Types in TypeScript

```ts
function repeat(text: string, times: number): string {
  return text.repeat(times);
}
// repeat("a", 3) → "aaa"
// repeat(1, 3)   → compile error
```

TypeScript adds **static types** on top of JavaScript. The compiler warns you when data does not match what your code expects, often before you run in the browser.

## Inference

```ts
const n = 42; // inferred: number
const xs = [1, 2, 3]; // number[]
const user = { id: 1, name: "Ada" }; // { id: number; name: string }
```

You do not need to annotate everything: TS **infers** types from values and control flow. You annotate when inference is too wide (implicit `any`) or when you want clearer API contracts.

## Explicit annotations

```ts
type User = { id: string; name: string };

function greet(user: User): string {
  return `Hello, ${user.name}`;
}
```

For public APIs or clearer intent, **annotations** (`: string`, `: User`) make the contract explicit for callers and for the compiler.

```json
{
  "compilerOptions": {
    "strict": true
  }
}
```

<Callout title="Strict mode">
With `strict: true` (recommended) you catch many bugs around `null`/`undefined` and questionable assignments.
</Callout>

## Strings with “shape”

Even when a value is a `string`, you may want **strict rules**: for example accept only decimal integers (`"42"`) and reject `"3.14"` or `"12px"`. That combines types (`number | null`) with runtime validation.
